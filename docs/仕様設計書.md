## 5. AI需要予測アルゴリズム設計

### 5.1 全体フロー

```
build_wholesale_forecast_rows(db, q)
  │
  ├─ 設定取得（Z係数・ABC閾値・予測モード）
  │
  ├─ build_abc_map(db)
  │   └─ sales_daily_agg 365日集計 → 累計比率 → A/B/Cランク判定
  │
  ├─ 商品一覧取得（products + stocks JOIN）
  │
  ├─ sales_daily_agg 180日データ一括取得 → jan別辞書構築
  │
  ├─ 気温データ取得（weather_data 90日）→ 平均気温算出
  │
  ├─ 気温感応度取得（temp_sensitivity テーブル）
  │
  └─ 商品ごとの予測計算ループ
      │
      ├─ Aランク（データ14日以上）
      │   ├─ holt_winters_forecast(daily_qtys, horizon=30)
      │   ├─ quantile_forecast(daily_qtys) → Q25/Q50/Q75/iqr_std
      │   ├─ get_temp_adj_factor() → 気温補正係数
      │   ├─ 30日予測 = HW × 気温補正 × 手動補正
      │   └─ 動的安全在庫 = Z × iqr_std × √リードタイム
      │
      └─ B/Cランク
          ├─ 直近28日移動平均
          └─ 固定安全在庫 = Z × σ × √リードタイム
```

### 5.2 ホルト・ウィンターズ指数平滑法（加法型）

```
パラメータ:
  α = 0.3  （レベル平滑化係数）
  β = 0.1  （トレンド平滑化係数）

初期化:
  L₀ = series[0]
  T₀ = series[1] - series[0]

更新式（各観測値 yₜ に対して）:
  Lₜ = α × yₜ + (1 - α) × (Lₜ₋₁ + Tₜ₋₁)
  Tₜ = β × (Lₜ - Lₜ₋₁) + (1 - β) × Tₜ₋₁

h期先予測:
  Ŷₜ₊ₕ = Lₜ + h × Tₜ    (max(0, ...))
```

### 5.3 動的安全在庫計算

```
IQR標準偏差:
  σ_IQR = (Q75 - Q25) / 1.349

動的安全在庫:
  SS = Z × σ_IQR × √L

  Z: サービスレベルZ係数（95%=1.65, 99%=2.33）
  L: リードタイム（日数）

推奨発注点:
  ROP = Q75 × L + SS
```

### 5.4 気温補正

```
モデル: 線形回帰
  y = a + b × T   (y=日次売上, T=気温)

気温補正係数:
  adj = b × (T_forecast - T_base) / daily_mean
  factor = clamp(1.0 + adj, 0.7, 1.3)  # ±30%制限

適用:
  forecast_adjusted = forecast_base × factor
```

### 5.5 ABC分析（累計比率法）

```
ABC分析計算:
  1. 過去365日の商品別売上金額を集計（sales_daily_agg × products.cost_price）
  2. 売上金額降順にソート
  3. 累計比率 = 累計売上 / 全体売上

ランク判定:
  累計比率 ≤ A閾値(70%):  → Aランク
  累計比率 ≤ B閾値(90%):  → Bランク
  その他:                 → Cランク

XYZ分析（変動係数）:
  CV = σ(週次売上) / μ(週次売上)
  CV < 0.25:  X（安定）
  CV < 0.75:  Y（普通）
  その他:     Z（不安定）
```

## 6. 自動スケジューラー設計

### 6.1 スケジューラー起動

```python
def start_scheduler():
    """Flaskアプリ起動時にバックグラウンドスレッドで開始"""
    thread = threading.Thread(target=_scheduler_loop, daemon=True)
    thread.start()

def _scheduler_loop():
    while True:
        now = datetime.now()
        # 毎日定時の発注チェック・期限チェック
        if _should_run_daily(now):
            _run_daily_tasks()
        # 毎月1日の月末取込
        if _should_run_month_end(now):
            _run_month_end_tasks()
        time.sleep(30)  # 30秒ごとにチェック
```

### 6.2 日次タスク詳細

```
run_order_check()
  ├─ 発注点以下の商品を抽出
  │   SELECT p.* FROM products p
  │   LEFT JOIN (SELECT jan, SUM(qty) FROM stocks GROUP BY jan) s ON s.jan = p.jan
  │   WHERE p.is_active = 1
  │     AND p.reorder_point > 0
  │     AND COALESCE(s.qty, 0) <= p.reorder_point
  │
  ├─ 混載グループの処理
  │   └─ mixed_lot_cases 達成まで order_pending に積む
  │
  └─ メール送信（仕入先別にグルーピング）

run_expiry_check()
  ├─ 賞味期限切れ間近の在庫を抽出
  │   SELECT s.*, p.expiry_alert_days
  │   FROM stocks s JOIN products p ON p.jan = s.jan
  │   WHERE s.expiry_date <= CURRENT_DATE + p.expiry_alert_days
  │     AND s.quantity > 0
  │
  └─ alert_logs に記録 + メールアラート送信
```

## 7. CSV取込設計

### 7.1 取込フロー

```
run_csv_import(setting_id, target_date)
  │
  ├─ DB から csv_import_settings を取得
  │
  ├─ ネットワーク共有フォルダ接続（net use）
  │   └─ UNCパス（\\server\share）対応
  │
  ├─ ファイル名パターン解決

  │   → '20260405_sales.csv'
  │
  ├─ CSV読み込み（encoding自動検出）
  │
  ├─ 行ごとの処理
  │   ├─ JAN正規化（13桁パディング）
  │   ├─ 数量変換（int）
  │   ├─ 日付正規化（YYYY-MM-DD）
  │   ├─ チェーン・店舗除外判定
  │   ├─ row_hash生成（jan + date + qty + store の SHA-256）
  │   └─ 重複チェック → INSERT sales_history
  │
  ├─ sales_daily_agg への集計更新
  │   INSERT INTO sales_daily_agg ...
  │   ON CONFLICT (jan, sale_dt) DO UPDATE SET qty = qty + ...
  │
```
